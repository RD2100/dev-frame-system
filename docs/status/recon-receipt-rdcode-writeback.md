## Core finding
The write-back is a **policy placeholder with zero execution code**. The bridge
is GET-only read on the client side; the one mutation channel
(`/api/t3/approval-response`) already enforces loopback + a human approve/reject
gate. So the cleanest, lowest-risk realization reuses that gate rather than
inventing a new endpoint or flipping the default write policy.

## Capability Matrix / Reuse decision
- mutation transport: REUSE `/api/t3/approval-response` (already gated, already
  the client's only write channel) — no new endpoint.
- human gate: REUSE the existing approve/reject + (optionally) the
  `/actions/execute` `confirm=execute` two-stage pattern.
- diff review surface (client, slice 2): REUSE native `review.getDiffPreview`.
- audit: REUSE `action-run.json` via `_start_execution_plan`.
- execution: BUILD NEW — a small governed write-back executor (apply a proposed
  single-file edit, path-safety confined to workspace root) invoked as a new
  action kind, mirroring the existing `code execute` CLI-subprocess pattern.

## Build-vs-Buy
- must_reuse: approval-response gate, action-run audit, native diff-preview UI.
- must_build_new: a governed write-back action kind in `_execution_plan_for_action`
  + a write executor with strict path safety (reject paths escaping workspace
  root, `.git`, `.env`, symlinks; mirror CodexPro's sensitive-path refusals) +
  policy/manifest updates (add the action kind to allowedActionKinds; bump
  `mutation_status` to `human-gated-local-actions`).
- must_NOT_change: `t3_client_shell` `writePolicy` stays `"read-only"` — the
  bridge default remains read-only; write-back is a per-action human-gated
  exception, NOT a mode flip. This keeps the schema intact and the "no silent
  write" guarantee.

## Integration Risk Table
- risk: write-back applies file changes / commits → high blast radius.
  type: security. severity: high. mitigation: human gate is mandatory
  (approve + confirm); writes confined to workspace root with sensitive-path
  refusal + symlink-escape block; every write produces an `action-run.json`
  audit; default stays read-only; no commit/push without explicit opt-in.
- risk: path traversal / writing outside workspace. type: security. severity:
  high. mitigation: resolve realpath, assert it is under the workspace root,
  reject `..`, absolute paths, `.git/`, `.env`, symlinks.
- risk: fork/public-repo boundary. type: license. severity: high. mitigation:
  Python write-back lives in the public repo (DevFrame governance); only the
  client diff-review wiring (slice 2) touches the fork, staying outside the
  public repo. `verify-public-snapshot.ps1` green; clean `.ai-bridge/` first.
- risk: schema drift. type: maintenance. severity: low. mitigation: update
  `visual_client_manifest.schema.json` + tests in lockstep; leave
  `t3_client_shell.writePolicy` unchanged.
