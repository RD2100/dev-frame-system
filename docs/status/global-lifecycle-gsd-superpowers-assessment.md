## 2. Comparison with GSD Pi

GSD Pi (Goal System Driven -- Pi edition) is a local coding agent runtime with: `.gsd` project memory, milestones/slices/tasks decomposition, auto-mode execution loop, failure forensics, and extension surface for tools/commands.

### 2.1 `.gsd` Project Memory

| GSD Pi concept | dev-frame-system equivalent | Gap |
|---------------|---------------------------|-----|
| `.gsd/` directory as project-local persistent state | `rules/`, `docs/`, capability inventory, rdgoal project contracts stored in project | dev-frame stores governance assets as plain files under project root instead of a single hidden directory. No single-file "project memory" blob -- instead uses runtime journal + digest + contract files. **Different architecture, same intent.** |
| Auto-loading of project memory on session start | `AGENTS.md` auto-read by agents; capability inventory loaded by bootstrap | dev-frame uses git-tracked plaintext docs rather than a tool-specific database. More portable, less tool-coupled. |
| `.gsd/auto.lock` for exclusive access | Conflict registry in SADP for parallel dispatch safety | dev-frame has conflict registry as a formal SADP concept but no filesystem-level lock mechanism. Minor gap. |

**Verdict:** dev-frame's equivalent is mature and provider-neutral. The `.gsd/` directory is a tool-specific optimization; dev-frame achieves the same through git-tracked governance files + runtime journal.

### 2.2 Milestones / Slices / Tasks

| GSD Pi concept | dev-frame-system equivalent | Gap |
|---------------|---------------------------|-----|
| Milestone briefs in `M###-CONTEXT.md` | Goal decomposition through SADP TaskSpec; project contracts define scope | No milestone-numbering convention. Goals are decomposed into slices through ai-workflow-hub `goal_store.py` (`slices` array with `slice-01`, `slice-02`). |
| Vertical slices as tracer bullets | `goal_store.py` slice tracking: `all_slices_passed()`, `slice_id = f"slice-{len(g['slices'])+1:02d}"` | Slice concept exists in ai-workflow-hub but is not exposed at the control-plane level. |
| Task dispatch | SADP `TaskSpec.json` + `TASKSPEC.md` dispatch to sub-agents | Fully implemented. Dispatch is the core of SADP. |

**Verdict:** Slice/task decomposition exists in ai-workflow-hub but is not surfaced in the control-plane visual state or action queue. Milestones are ad-hoc (docs/status/ stage tracking) rather than a formal convention.

### 2.3 Auto Mode

| GSD Pi concept | dev-frame-system equivalent | Gap |
|---------------|---------------------------|-----|
| Unattended execution loop | `auto_execute` decision mode in rdgoal dispatches reversible work freely | dev-frame has decision-based auto dispatch, not a continuous auto-mode loop. The `--Prepare --Dashboard` -> `--Execute` pattern in the bootstrap wrapper requires human review between stages. |
| Checkpoint/snapshot before each step | `snapshot_execute` mode creates rollback snapshots before destructive ops | Implemented. |
| Auto-loop with break conditions | Not implemented | dev-frame stops after each decision; no "continue until blocked" loop. Intentionally conservative -- auto mode is a GSD-style optimization that conflicts with dev-frame's reviewer-first design. |

**Verdict:** dev-frame intentionally avoids unattended auto-mode loops. The `auto_execute` decision mode dispatches single reversible operations, not continuous chains. This is a design choice tied to the mandatory reviewer node in SADP.

### 2.4 Failure Forensics

| GSD Pi concept | dev-frame-system equivalent | Gap |
|---------------|---------------------------|-----|
| `.gsd/activity/*.jsonl` activity traces | `.devframe-runtime/rdgoal-events.jsonl` journal | Implemented. |
| `.gsd/journal/YYYY-MM-DD.jsonl` run logs | ai-workflow-hub run directories with `execution_log` | Partial -- only in ai-workflow-hub path, not universal. |
| `.gsd/metrics.json` | Not implemented | No central metrics collection. Gap. |
| Post-mortem tooling | runtime `failure-analysis.md` output from the ai-workflow-hub finalizer plus the Failure Pattern Library in dispatch-model-profiles.md | Present but fragmented across packages. No single "forensics" command or tool. |

**Verdict:** The pieces exist (journal, failure analysis, pattern library) but are not unified into a single forensics surface. GSD's `.gsd/activity/*.jsonl` is more granular per-step; dev-frame's journal is coarser per-decision.

### 2.5 Extension Surface

| GSD Pi concept | dev-frame-system equivalent | Gap |
|---------------|---------------------------|-----|
| Extension SDK for tools/commands/hooks | Command worker adapter (`rdgoal worker --command`); ai-workflow-hub adapter (`--aihub-go`) | Narrow. Only worker substitution, not general tool/command/hook extension. |
| Custom TUI components | Dashboard endpoints (`/client-manifest.json`, `/t3-bridge.json`) plus T3 Code bridge | Read-only dashboard; T3 bridge is a reuse path, not an extension SDK. |
| Plugin registry | None | No plugin system. Extension is through worker adapters and the T3 bridge contract. |

**Verdict:** dev-frame's extension surface is intentionally minimal. The philosophy is: workers are replaceable; the control plane is not pluggable. GSD's extension surface is broader because GSD is a coding agent itself; dev-frame is a governance layer above replaceable agents.

## 3. Comparison with Superpowers

"Superpowers" describes a methodology pattern: skill-triggered discipline sequences (brainstorm -> spec -> plan -> implement -> review -> finalize), with TDD as a foundational practice and methodology-discipline skills that enforce process.

### 3.1 Skill-Triggered Methodology

| Superpowers concept | dev-frame-system equivalent | Gap |
|--------------------|---------------------------|-----|
| Skills triggered by user intent patterns | `@go`, `@go read`, `@go edit`, `@go risky` trigger `agent-acceptance` skill | Only one trigger pattern (`@go`). No general skill-trigger framework -- each trigger is hardcoded. |
| Multiple methodology skills (TDD, spike, etc.) | Only `agent-acceptance` exists as a methodology-discipline skill | No TDD skill, no spike skill, no spec-writing skill as triggerable disciplines. |
| Skill registry/discovery | Skill intake schemas exist (`local-skill-intake-record.schema.json`) but no runtime discovery | Formal schema for intake but no runtime skill directory or discovery mechanism. |

**Verdict:** dev-frame has a single methodology skill (`agent-acceptance`) with one trigger (`@go`). Superpowers' "skill-triggered methodology" is a broader vision of many discipline skills each with their own triggers. dev-frame has the intake schema ready but no runtime.

### 3.2 TDD Discipline

| Superpowers concept | dev-frame-system equivalent | Gap |
|--------------------|---------------------------|-----|
| Red-green-refactor loops | Not present | No TDD skill, no test-first workflow enforcement. SADP has a tester role but it validates after implementation, not before. |
| Test generation before implementation | Not present | Tests are generated as evidence collection, not as design. |

**Verdict:** TDD as a formal methodology discipline is absent from dev-frame. The tester role in SADP serves evidence collection, not test-first design. This is a methodology gap, not a tooling gap.

### 3.3 Brainstorm / Spec / Plan / Implement / Review / Finalize

| Superpowers concept | dev-frame-system equivalent | Gap |
|--------------------|---------------------------|-----|
| Brainstorm phase | Not formalized | No brainstorming methodology. `recommend_execute` in rdgoal covers direction choices but not ideation. |
| Spec phase | Project contracts, TaskSpec, goal decomposition | Spec exists as project-scope contracts and machine-readable TaskSpecs, but not as a separate "spec writing" phase. |
| Plan phase | SADP Goal Agent decomposition; `devframe go` with `--Prepare` | Implemented. |
| Implement phase | SADP executor/fixer role | Implemented. |
| Review phase | SADP mandatory reviewer role (separate session/model) | Implemented and enforced. |
| Finalize phase | SADP finalizer: `go_evidence.py finalize`, plus runtime `final-report.md` artifacts under the evidence directory | Implemented. |

**Verdict:** The SADP state machine (`human_gate -> executor/fixer -> tester -> reviewer -> finalizer`) covers most of the Superpowers lifecycle. Missing: brainstorm (ideation), spec (formal spec writing as a phase). The core *implement -> review -> finalize* chain is mature.

## 5. Concrete First Two Slices

### Slice 1: Methodology Skill Framework

**Goal:** Generalize the `agent-acceptance` trigger pattern into a reusable methodology skill framework so new discipline skills (TDD, spike, spec-writing) can be added as standalone triggerable skills.

**Scope:**
- Define a `methodology-skill` contract: trigger patterns, discipline sequence, required evidence, role boundaries.
- Refactor `agent-acceptance` SKILL.md as the first conforming methodology skill (no behavior change).
- Implement a light skill registry in the control plane that maps trigger patterns to skill paths.
- Expose registered skills in visual-state output (`skills` field under `project`).
- Add to bootstrap in a later slice: a `tools/skills/` directory with
  `agent-acceptance/SKILL.md` as the first entry.

**Allowed files:** `packages/control-plane/control_plane/skill_registry.py` (new), `schemas/methodology-skill.schema.json` (new), `docs/agent-runtime/methodology-skills.md` (new), minor change to `visual_state.py`.

**Acceptance:**
- `devframe visual-state` output includes a `skills` field listing registered methodology skills.
- `agent-acceptance` continues to work unchanged.
- Future target: a new methodology skill can be registered by adding a
  conforming `SKILL.md` to `tools/skills/`.

### Slice 2: TDD Discipline Skill

**Goal:** Add a planned `tdd` methodology skill that enforces test-first
workflow (red-green-refactor) within the existing SADP framework.

**Scope:**
- Write `tdd/SKILL.md` conforming to the methodology-skill contract from Slice 1.
- Define the TDD discipline sequence: `write-test -> verify-red -> implement -> verify-green -> refactor -> verify-green`.
- Map TDD roles onto SADP roles: test author = executor/fixer (test-first pass), refactor = same executor/fixer (implementation pass), reviewer validates both passes.
- Define TDD evidence requirements: `test-output-red.md`, `test-output-green.md` with before/after exit codes.
- Register TDD as a methodology skill triggered by `@tdd` or explicit `--discipline tdd` flag.

**Allowed files:** planned `tools/skills/tdd/SKILL.md`, plus minor references
in `packages/control-plane/control_plane/skill_registry.py`.

**Acceptance:**
- `@tdd` triggers the TDD methodology.
- Executor must show test failure (red) before implementation evidence is accepted.
- Reviewer can distinguish TDD runs from standard runs via methodology tag in evidence.

## 7. Recommendation

**Recommendation: Build the methodology skill framework (Slice 1) and TDD discipline (Slice 2). Do not adopt GSD or Superpowers as external tooling.**

Rationale:

1. dev-frame-system already has a more rigorous lifecycle than GSD Pi or Superpowers in the areas where it overlaps: decision engine, SADP state machine, P0-P3 verification gates, evidence chain, mandatory reviewer separation, and bootstrap distribution.

2. The two concrete gaps are: (a) no general methodology skill framework (only one hardcoded skill), and (b) no TDD discipline. Both can be fixed with minimal surface additions without importing external dependencies or coupling to any specific agent runtime.

3. GSD Pi's auto-mode loop and Superpowers' agent-native UI are intentionally excluded -- they conflict with dev-frame's governance-first, provider-neutral design.

4. The recommended path is dev-frame-native throughout: keep the battle-tested governance layer, build the missing methodology surface within it, borrow only lightweight patterns (JSONL forensics granularity, skill-installer intake) when they don't create coupling.
