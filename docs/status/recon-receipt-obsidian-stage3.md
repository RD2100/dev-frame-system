## Reuse Candidate List
- candidate: Obsidian URI
  - source: https://obsidian.md/help/uri
  - exact_scope_to_reuse: open generated notes by vault and file path.
  - expected_adapter_work: generate percent-encoded open links.
  - blocking_constraints: vault name may differ from folder name; expose as local convenience only.
  - decision: reuse now.
- candidate: Obsidian Advanced URI
  - source: https://github.com/Vinzent03/obsidian-advanced-uri
  - exact_scope_to_reuse: richer future navigation and command automation.
  - expected_adapter_work: optional URI formatter.
  - blocking_constraints: plugin install required; not needed for stage 3 minimal.
  - decision: defer.
- candidate: Obsidian Local REST API
  - source: https://github.com/coddingtonbear/obsidian-local-rest-api
  - exact_scope_to_reuse: authenticated read/write, patch by heading/frontmatter, search, open note.
  - expected_adapter_work: small optional client and auth config.
  - blocking_constraints: user must install plugin and provide API key/cert; not suitable for default test path.
  - decision: adapt later for stage 2 live bridge, not required for this slice.
- candidate: Custom Obsidian plugin
  - source: https://docs.obsidian.md/
  - exact_scope_to_reuse: none.
  - expected_adapter_work: TypeScript plugin, UI, release packaging.
  - blocking_constraints: too much maintenance before CLI pipeline stabilizes.
  - decision: reject for now.

## Build-vs-Buy Decision
- must_reuse:
  - Obsidian URI for open links.
  - Dataview/Bases-compatible frontmatter for in-vault views.
- should_adapt:
  - Local REST API in a future live bridge slice.
- can_spike:
  - Advanced URI links for headings and commands.
- must_build_new:
  - Minimal note schema, managed block, dashboard template, and report fields.
- rationale: The repo already owns the research pipeline and safety boundary. Obsidian owns the UI and runtime API surface. The smallest stage 3 slice should write better Markdown, not build a plugin.
