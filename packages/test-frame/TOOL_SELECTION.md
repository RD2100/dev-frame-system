## 二、各领域详细选型

### 2.1 测试管理 + 接口测试 + Mock

#### 推荐方案：MeterSphere + Apifox 组合

| 维度 | MeterSphere | Apifox |
|------|-------------|--------|
| **定位** | 一站式持续测试平台 | API设计+调试+Mock一体化 |
| **开源** | 核心开源 (GPL 3.0) | 闭源，免费版可用 |
| **部署** | 自托管/SaaS | SaaS为主，私有化付费 |
| **测试管理** | ★★★★★ 完整用例管理+测试计划 | ★★☆☆☆ 基础接口级用例 |
| **接口测试** | ★★★★★ JMeter引擎+多协议 | ★★★★☆ HTTP/WS/gRPC/Dubbo |
| **Mock服务** | ★★★☆☆ 基础响应模拟 | ★★★★★ 智能Mock+自定义脚本 |
| **性能测试** | ★★★★★ 集成JMeter | ☆☆☆☆☆ 无 |
| **API设计** | ☆☆☆☆☆ 无 | ★★★★★ 可视化编辑+Schema |
| **CI/CD集成** | ★★★★☆ Jenkins/GitLab插件 | ★★★☆☆ CLI+API |
| **学习曲线** | 较陡 | 较平（类Postman） |
| **定价** | 开源免费/专业版付费 | 免费版/专业版¥29/人/月起 |

**组合策略**：
- Apifox 负责 API 设计 → Mock → 日常调试（前端驱动）
- MeterSphere 负责 测试管理 → 自动化执行 → 性能测试 → 报告（QA驱动）
- 通过 OpenAPI/Swagger 格式打通接口定义，实现工具链协同

**替代方案**：
- **Postman**：全球最流行，但团队协作付费($14/人/月起)，国内访问慢，Mock基础
- **SoapUI**：SOAP/REST双支持，但界面老旧，开源版功能有限

### 2.2 Android App 自动化

#### 推荐：Airtest + Poco（全量回归）+ Maestro（冒烟测试）

| 维度 | Airtest+Poco | Maestro | Appium | UIAutomator2 |
|------|-------------|---------|--------|-------------|
| **脚本语言** | Python | YAML | 多语言 | Java/Python |
| **学习曲线** | 低 | **极低** | 高 | 中等 |
| **定位方式** | 图像+控件 | 控件 | 控件(+图像插件) | 控件 |
| **执行速度** | 中等 | 快 | 慢 | **最快** |
| **稳定性** | 中等 | **高** | 中等偏弱 | 高 |
| **CI/CD集成** | 中等 | **极简** | 复杂 | 简单 |
| **跨平台** | Android/iOS/Win/Web | Android/iOS | Android/iOS | **仅Android** |
| **游戏支持** | **最强** | 弱 | 弱 | 弱 |
| **复杂逻辑** | 强 | 弱(YAML限制) | **最强** | 强 |
| **定价** | 免费 | 免费+云收费 | 免费 | 免费 |
| **集成成本** | 中 | **极低** | 高 | 低 |

**分层策略**：
```
冒烟测试层（快速反馈）    → Maestro (YAML, 5分钟写一个流)
   ↓ 通过后
核心回归层（稳定性验证）  → Airtest+Poco (Python, 图像+控件双模式)
   ↓ 通过后
专项测试层（性能/兼容）   → Airtest + WeTest云真机
```

**选Airtest+Poco而非Appium的理由**：
1. 图像识别兜底控件无法覆盖的场景（游戏/动态UI）
2. Poco对Unity/Cocos2d游戏引擎支持是独有能力
3. 国内社区活跃，中文文档完善
4. Python脚本比Appium的Client-Server架构更轻量

**选Maestro做冒烟的理由**：
1. YAML声明式，5分钟写一个测试流，维护成本最低
2. 内置智能等待，flaky rate极低
3. CI集成一条命令 `maestro test flow.yaml`
4. 与Airtest互补不冲突

### 2.3 微信小程序自动化

#### 推荐：miniprogram-automator（唯一选择）

| 维度 | 评估 |
|------|------|
| **核心能力** | 官方出品，操控微信小程序运行时；页面跳转/组件查找/元素操作/Mock wx API |
| **优势** | 唯一能直接操控小程序真机运行时的工具；可访问page.data/组件实例；支持mockWxMethod模拟登录支付 |
| **劣势** | 仅支持微信小程序；依赖微信开发者工具(500MB+)；CI环境需预装开发者工具；Docker化困难 |
| **集成成本** | 中等偏高，需安装开发者工具+配置自动化端口+编写启动脚本，预计3-5天 |
| **定价** | 完全免费 (MIT) |

**无替代方案**：微信小程序运行在微信客户端定制JSCore中，非标准浏览器环境，`wx.xxx` API是微信客户端注入的，其他工具无法触及。

### 2.4 H5/uni-app 自动化

#### 推荐：Playwright

| 维度 | Playwright | Cypress | Selenium |
|------|-----------|---------|----------|
| **跨浏览器** | Chromium/Firefox/WebKit | 仅Chromium | 全浏览器 |
| **自动等待** | 内置完善 | 内置完善 | 需手动WebDriverWait |
| **录制生成** | Codegen(优秀) | 无内置 | Selenium IDE(一般) |
| **调试体验** | Trace Viewer(优秀) | Time Travel(最佳) | 差(靠截图) |
| **CI/CD集成** | 优秀(一键安装) | 优秀(Docker) | 成熟但繁琐 |
| **网络Mock** | route.fulfill | cy.intercept | 有限 |
| **Docker支持** | 官方镜像 | 官方镜像 | 社区镜像 |
| **学习曲线** | 低 | 低 | 中高 |
| **定价** | 免费 | 免费/付费 | 免费 |

**选Playwright而非Cypress的理由**：
1. 三引擎支持含WebKit（模拟iOS Safari），Cypress仅Chromium
2. 网络拦截Mock能力更强，可模拟API响应
3. Codegen录制大幅降低编写成本
4. CI友好：`npx playwright install`一键安装

**选Playwright而非Selenium的理由**：
1. 自动等待机制完善，几乎不写wait()
2. Trace Viewer调试体验远超Selenium截图
3. 执行速度更快
4. 新项目无理由选Selenium

### 2.7 报告聚合

#### 推荐：Allure（主报告）+ MeterSphere（测试管理报告）

| 维度 | Allure | MeterSphere内置 | ReportPortal | ExtentReports |
|------|--------|----------------|--------------|---------------|
| **多源聚合** | ★★★★☆ | ★★★☆☆ | ★★★★★ | ★★☆☆☆ |
| **趋势分析** | TestOps付费 | 内置基础 | AI驱动最强 | 几乎无 |
| **缺陷关联** | 注解关联 | 内置+JIRA同步 | AI自动聚类 | 无 |
| **报告美观** | ★★★★★ | ★★★☆☆ | ★★★☆☆ | ★★★★☆ |
| **部署复杂度** | 低 | 中 | 高 | 最低 |
| **多框架适配** | 最广 | 平台内 | 广 | Java/.NET |
| **开源免费** | 社区版免费 | 社区版免费 | 开源版完整 | 社区版免费 |

**选Allure的理由**：
1. 报告美观度业界标杆
2. 适配Airtest/Maestro/Playwright/pytest等所有框架
3. 轻量级，CI集成简单（allure-results目录）
4. 社区活跃，文档完善

**不选ReportPortal的理由**：部署复杂（需ES/PG/RabbitMQ/MinIO），资源消耗大(8GB+)，对当前规模过重。

### 2.9 缺陷归因

#### 推荐：自研轻量规则引擎

**不选ReportPortal AI的理由**：
1. 部署过重（8GB+内存，4个中间件依赖）
2. AI黑盒，归因逻辑不可控不可解释
3. 当前测试规模不需要AI自动聚类

**自研规则引擎设计**：
- 基于规则的归因（非AI），逻辑透明可解释
- 规则可配置：错误类型→模块映射、堆栈关键词→组件映射、失败模式→根因映射
- 输出归因报告：失败用例→疑似根因→建议修复方向
- 轻量实现：Python规则文件 + 简单匹配引擎

## 三、工具能力边界总结

| 工具 | 已具备能力 | 我们不做的 |

| Airtest+Poco | UI自动化脚本、图像识别、控件识别 | — |
| Maestro | 冒烟测试YAML流、智能等待 | — |
| miniprogram-automator | 小程序操控、wx API Mock | — |
| Playwright | H5自动化、跨浏览器、录制、网络Mock | — |
| WeTest | 云真机、兼容性测试、小程序测试 | — |
| Sentry | 错误追踪、性能监控、Release追踪 | — |
| Bugly | 移动端崩溃监控、小程序崩溃 | — |

### 我们只做的"胶水层"

| 胶水模块 | 职责 | 不做的 |
|---------|------|--------|
| 配置统一层 | 统一环境/设备/账号配置 | 不替代各工具自身配置 |

| 缺陷归因规则层 | 规则匹配失败→根因 | 不做AI黑盒归因 |
| CI/CD调用脚本 | 封装各工具的CI调用方式 | 不替代CI/CD平台 |

## 五、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| miniprogram-automator CI环境难搭建 | 小程序自动化无法CI化 | 预装开发者工具的专用构建机 |
| Airtest图像识别不稳定 | 回归测试flaky | 优先Poco控件模式，图像仅兜底 |

| Apifox闭源数据在云端 | 数据安全顾虑 | 导出OpenAPI格式本地备份 |
| Maestro YAML复杂逻辑受限 | 无法覆盖复杂场景 | 复杂场景交给Airtest+Poco |
