P0：derive_projection() 会把已有的 missing-context blocked 场景错误推导成 ready
missing-context.json 明确表示：work_item.status = blocked，input_context_artifact_id = artifact-ctx-missing，decisions = []，并且既有 projection 是 computed_status = blocked、allowed_actions = ["view", "escalate"]。见 missing-context.json:10-17、missing-context.json:40-41、missing-context.json:56-68。
但 derive_projection() 当前逻辑在没有相关 decision 时直接落到：


computed_status = "ready"：review_governance_validator.py:322-324


allowed_actions = ["view", "execute", "edit"]：review_governance_validator.py:351-358


这会把一个缺失输入上下文、应当 blocked 的 work item 推成 ready，并开放 execute/edit。这个不是保守投影，而是发明 readiness 权威，违反“不发明权威”的核心原则。
P1：blocked_reason 没有稳定从真实 payload.blocked_reasons 推导
blocked.json 中 gate decision 的 payload 明确有：
"blocked_reasons": ["tool boundary violation: playwright_bridge wrote outside agent-runtime"]，并且 fixture projection 也使用了这个具体原因。见 blocked.json:101-121、blocked.json:138-145。
但 derive_projection() 遇到 blocked decision 时使用第一个 blocked decision；在该 fixture 中第一个是 review blocked，没有 payload，于是会返回默认 "blocked"，而不是 gate payload 中的具体 blocked reason。问题在 review_governance_validator.py:306-312。
这不影响当前测试，因为 test_derive_projection_blocked_from_blocked_decision 只断言 status，没有断言 blocked_reason。见 test_review_governance_kernel.py:554-558。
P1：latest_decision_id 的推导不是按原始 decision 顺序
当前代码先把 review 和 gate 分开收集，再拼成：
wi_decisions = wi_reviews + wi_gates
然后取 wi_decisions[-1] 作为 latest。见 review_governance_validator.py:276-288。
这会破坏原始 decisions 数组的时间顺序。只要出现“gate 后又追加 review”或其他交错记录，latest_decision_id 就可能错误。测试只覆盖了 success.json 中 review 在前、gate 在后的简单顺序。见 test_review_governance_kernel.py:582-585。
P1：状态推导使用“历史 any”而非“当前有效状态”
当前状态判断是：


任意 human_required → blocked


任意 blocked → blocked


任意 insufficient_evidence → insufficient_evidence


任意 gate pass → completed


任意 review pass → reviewing


见 review_governance_validator.py:302-324。
这意味着一个早期 blocked / insufficient_evidence 记录即使后来被 gate pass 解决，也会永久压过后续 pass。若 decisions 是追加式事实日志，当前投影应基于最新有效 gate/review 结果或明确的状态转移规则，而不是简单 any()。

derive_projection 的状态推导逻辑是否正确？
不正确。至少 missing-context 场景会 false-ready；历史 blocked 也可能错误覆盖后续 pass。


allowed_actions 映射是否合理？
映射本身作为初版可以接受：blocked → view/escalate，insufficient_evidence → view/add_evidence/review，completed → view/archive。但由于 computed_status 可被错误推成 ready，当前会错误开放 execute/edit，所以不能通过。


derive_projection 是否保持了“不发明权威”的原则？
没有。把缺失上下文的 blocked packet 推导为 ready，就是发明了可执行状态。
